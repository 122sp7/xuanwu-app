## 2. 功能地圖

Xuanwu Notion 對齊的功能位於 **Wiki-Beta** 模組下：

| 功能 | 對應 Notion 概念 | 路徑 | 典型用途 |
|---|---|---|---|
| **Pages** | Page tree（階層頁面） | `/wiki-beta/pages` | 用樹狀結構組織知識、會議紀錄、文件 |
| **Libraries** | Database（結構化資料表） | `/wiki-beta/libraries` | 管理任務清單、技能清冊、文件索引 |
| **Documents** | Knowledge documents（知識文件庫） | `/wiki-beta/documents` | 上傳 PDF/圖片供 AI 問答 |
| **RAG 問答** | AI 知識查詢 | `/wiki-beta/rag-query` | 對已上傳文件進行自然語言問答 |

## 3. Pages 使用指南

### 3.1 什麼是 Pages？

Pages 讓你用**樹狀結構**組織知識內容，類似 Notion 的 Page tree。

```
我的知識庫（根頁面，使用者建立）
├── 產品設計
│   ├── 使用者訪談紀錄
│   └── 設計決策
└── 技術架構
    ├── 資料模型
    └── API 設計
```

每個頁面可以有多個子頁面，形成可無限延伸的層級結構。系統不會自動建立任何根頁面，所有頁面都由使用者手動建立。

### 3.2 進入 Pages

1. 點選左側 App Rail 中的 **Wiki Beta**。
2. 在 Wiki-Beta 導覽列選擇 **Pages**。
3. 你會看到目前帳號下所有根頁面的樹狀列表。

### 3.3 建立頁面

**建立根頁面**（最上層）：

1. 進入 `/wiki-beta/pages`。
2. 點選 **新增頁面** 按鈕（通常在頁面右上角）。
3. 輸入頁面標題（最長 120 字元）。
4. 點選 **建立** 完成。

> **你會看到**：新頁面出現在樹狀列表中，slug 自動從標題生成。

**建立子頁面**（放在某個頁面底下）：

1. 在頁面樹中找到目標父頁面。
2. 點選該頁面右側的 **`+`** 圖示。
3. 輸入子頁面標題。
4. 點選 **建立** 完成。

> 子頁面自動歸屬於所選的父頁面，並繼承相同的帳號範圍（account scope）。

### 3.4 更名頁面

1. 在頁面樹中找到目標頁面。
2. 點選頁面名稱旁的 **編輯（鉛筆）** 圖示，或雙擊頁面名稱。
3. 輸入新標題。
4. 按下 Enter 或點選 **儲存** 完成。

> **你會看到**：頁面名稱更新，slug 同步更新並自動去重。

### 3.5 移動頁面

1. 在頁面樹中找到要移動的頁面。
2. 拖曳頁面到目標位置，或點選 **移動** 選項。
3. 選擇新的父頁面（或選擇「移至最上層」升為根頁面）。
4. 確認移動。

> **注意**：系統會自動阻擋形成循環的移動（例如把父頁面移動到其子頁面下），並顯示「無法移動：會造成循環結構」的提示。

### 3.6 常見問題

**Q：為何看不到我建立的頁面？**

1. 確認目前帳號是否正確（App Rail 右上角）。
2. 確認頁面狀態是否為 `active`（封存的頁面不顯示在樹中，請找管理員確認）。

**Q：為何不能將頁面移到某個位置？**

該操作可能造成循環結構（頁面成為自己的子孫），系統會自動阻擋。請選擇其他目標位置。

**Q：標題可以重複嗎？**

標題可以重複，但 slug（URL 用的識別碼）在同一層下會自動去重（例如 `my-page`、`my-page-2`）。

## 4. Libraries 使用指南

### 4.1 什麼是 Libraries？

Libraries 讓你用**結構化的欄位（Fields）和記錄（Records）**管理資料，類似 Notion 的 Database。

適合用於：
- ✅ 任務清單（欄位：標題、狀態、負責人、截止日）
- ✅ 技能清冊（欄位：技能名稱、等級、類別）
- ✅ 文件索引（欄位：文件名稱、類型、相關頁面連結）

### 4.2 進入 Libraries

1. 點選左側 App Rail 中的 **Wiki Beta**。
2. 在 Wiki-Beta 導覽列選擇 **Libraries**。
3. 你會看到目前帳號下所有 Libraries 的列表。

### 4.3 建立新 Library

1. 進入 `/wiki-beta/libraries`。
2. 點選 **新增資料庫** 按鈕。
3. 輸入 Library 名稱（最長 80 字元）。
4. 點選 **建立** 完成。

> **你會看到**：新 Library 出現在列表中，預設沒有欄位。下一步需要新增欄位才能輸入記錄。

### 4.4 新增欄位（Fields）

Library 建立後預設無欄位。先新增欄位定義資料結構，再輸入記錄。

**支援的欄位型別**：

| 型別 | 說明 | 範例用途 |
|---|---|---|
| `title` | 標題文字（主欄位） | 任務名稱、文件標題 |
| `text` | 多行文字 | 說明、備註 |
| `number` | 數字 | 優先級、評分 |
| `select` | 單選清單 | 狀態（待處理/進行中/完成）、類別 |
| `relation` | 跨紀錄引用 | 關聯其他 Library 記錄或頁面 |

**新增欄位步驟**：

1. 進入目標 Library。
2. 點選 **新增欄位（+）**。
3. 填入欄位標籤（Label，顯示給使用者看的名稱）。
4. 選擇欄位型別。
5. 若為 `select` 型別，輸入可選項目清單（例如：待處理、進行中、完成）。
6. 若此欄位為必填（不可留空），勾選 **必填（Required）**。
7. 點選 **儲存** 完成。

### 4.5 新增記錄（Records）

記錄對應 Notion 資料庫中的一列（Row）。

1. 進入目標 Library。
2. 點選 **新增記錄（+）** 或 **New Record**。
3. 填入各欄位的值（標記為必填的欄位不可留空）。
4. 點選 **建立** 完成。

> **你會看到**：記錄出現在 Library 列表中。若缺少必填欄位，系統會顯示「請填入 {欄位名稱}」的提示並阻止建立。

### 4.6 常見問題

**Q：為何無法建立記錄？**

確認所有標示為「必填」的欄位都已填入值。如果 Library 尚未新增任何欄位，也需要先新增至少一個欄位。

**Q：為何 `select` 欄位沒有選項可選？**

建立或編輯欄位時，需在 `options` 欄位輸入可選項目。若欄位已存在但無選項，請重新編輯欄位新增選項。

**Q：如何修改欄位順序或刪除欄位？**

進入 Library 後，點選欄位標題旁的設定圖示（⚙），可以重新排序或刪除欄位。注意：刪除欄位後相關記錄的對應值也會一起刪除。

## 5. Documents 快速建立（Notion-like 入口，planned）

### 5.1 功能說明

Wiki-Beta 側邊欄 Documents 項目右側的 **`+`** 圖示，提供快速建立頁面或 Library 的入口，對齊 Notion 在側邊欄快速新增內容的使用體驗。

> **目前狀態**：本功能規劃中（Phase 5），尚未正式上線。上線後可在此處操作。

### 5.2 操作流程（上線後）

1. 在 Wiki-Beta 側邊欄，找到 **Documents** 項目。
2. hover 後點選 Documents 右側的 **`+`** 圖示。
3. 在彈出選單選擇：
   - **新增頁面**：建立新的階層頁面（等同進入 Pages 後新增）
   - **新增資料庫**：建立新的 Library（等同進入 Libraries 後新增）
4. 填入名稱後確認建立。

> **你會看到**：成功後顯示 toast 提示，並自動導向新建立的頁面或 Library。

## 6. RAG 知識問答

RAG（Retrieval-Augmented Generation）讓你對已上傳的知識文件進行**自然語言問答**，AI 會從文件中找出相關片段並生成回答。

### 6.1 前置準備：上傳知識文件

1. 進入 `/wiki-beta/documents`。
2. 將檔案拖入 Upload 區塊，或點擊選檔。
3. 確認格式為 `.pdf`、`.tiff`、`.png` 或 `.jpg`。
4. 點選 **上傳並啟動解析**。
5. 等待背景 pipeline 完成：
   - `processing` → 解析中，請稍候
   - `ready` → ✅ 可進行 RAG 問答
   - `error` → 解析失敗，請重新上傳

> 文件解析通常需要 30 秒至數分鐘，視文件大小而定。

### 6.2 進行 RAG 問答

1. 進入 `/wiki-beta/rag-query`。
2. 在輸入框輸入問題（例如：「這個專案的主要風險是什麼？」）。
3. 設定 `top_k`（決定參考幾筆相關片段，預設 5，保留即可）。
4. 點選 **送出查詢**。
5. 查看結果：
   - **Answer**：AI 生成的回答
   - **Citations**：引用的原始文件片段與來源

> **若 Answer 無 Citations**：確認相關文件的 `rag status` 為 `indexed`（不是 `pending`）。若非，可至 RAG Reindex 頁面對該文件執行手動重整後再試。

## 7. 操作建議

### 知識組織策略

| 場景 | 建議工具 | 原因 |
|---|---|---|
| 撰寫說明文件、會議紀錄、設計決策 | Pages | 樹狀結構適合層級化、敘述性內容 |
| 管理任務、清單、索引 | Libraries | 結構化欄位讓篩選/排序更方便 |
| 分析或查找文件內容 | Documents + RAG 問答 | AI 直接從文件提取回答 |

### 日常使用流程

1. **整理知識**：用 Pages 建立文件結構，將相關頁面組織為子頁面。
2. **管理任務資料**：用 Libraries 建立結構化清單（任務、文件索引、技能清冊等）。
3. **AI 知識查詢**：先在 Documents 確認文件狀態為 `ready`，再進行 RAG 問答。
4. **workspace 視角**：若需查看特定工作區內容，使用 `workspaceId` 篩選；需全帳戶瀏覽時清除篩選即可。

## 8. 權限與資料範圍

| 概念 | 說明 |
|---|---|
| account（帳號） | 資料主範圍。Pages、Libraries、Documents 均屬於你的帳號，不與其他帳號共用。 |
| workspace（工作區） | 同帳號下的視角分組。workspace 是篩選用途，不是資料隔離邊界。 |
| 跨帳號資料 | 你只能存取自己帳號下的資料；跨帳號資料不可見。 |

> **重要**：workspace 的設計是「視角分組」，不是「資料隔離」。不帶 workspace 篩選時，你可以看到帳號下所有 workspace 的資料，這是正常行為，不代表資料外洩。
